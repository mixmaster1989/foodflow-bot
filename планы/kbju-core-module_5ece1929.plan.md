### Новый модуль ядра КБЖУ: цели

- **Главный приоритет (A)**: стабильные и консистентные КБЖУ для одного и того же продукта во времени (минимум "гуляющих" значений).
- **Второй приоритет (B)**: существенное снижение количества обращений к ИИ/интернету через умный кэш и переиспользование уже проверенных данных.
- **Интеграция**: сначала отдельный сервис/модуль, который можно вызывать из существующих мест (обёртка над `NormalizationService` и `AIService`), потом постепенная замена прямых вызовов.
- **Третий компонент**: защитник-анализатор (anomaly guard), который отлавливает аномальные КБЖУ и просит подтверждение/пересчёт.

### Архитектура нового модуля

- **Расположение (предложение)**: новый модуль `kbju_core` или `services/kbju_core.py` (название можем уточнить позже).
- **Публичный фасад модуля** (условные методы):
  - `get_product_nutrition(query: ProductQuery) -> NutritionResult`
  - `recheck_product_nutrition(product_id / fingerprint) -> NutritionAuditResult`
  - `analyze_nutrition_anomalies(nutrition: NutritionResult, context) -> AnomalyDecision`
- **Слой кэша**:
  - Хранение "эталонных" КБЖУ в БД (предпочтительно в отдельной таблице типа `canonical_products` или `kbju_cache`), чтобы не завязываться на сырые логи.
  - Ключи кэша: нормализованное название + возможно `base_name` + опционально страна/источник.
  - Значения: `calories, protein, fat, carbs, fiber`, источник (AI/label/manual), версия и timestamp проверки.
- **Слой перепроверки**:
  - Работает поверх уже существующих данных в БД (`products`, `consumption_logs`, возможно `label_scans`).
  - Периодически или по запросу поднимает кандидатов на перепроверку (неточные, старые, сильно расходящиеся с новыми данными).
  - Дергает ИИ/интернет, но результат записывает в кэш как новую версию эталона (с историей изменений).
- **Anomaly Guard**:
  - Получает на вход предлагаемые КБЖУ (из ИИ/чека/фото) и сравнивает их:
    - с эталоном из кэша (если есть),
    - с типичными диапазонами по категории ("масла", "фрукты", и т.п.),
    - с относительным контекстом дня пользователя (чтобы не выстрелить по каждому расхождению).
  - Результат: `OK`, `WARN_USER_WITH_TEXT`, `RECHECK_WITH_AI`, или `BLOCK_AND_ASK_MANUAL`.

### Детализация компонентов

#### 1. Кэш КБЖУ (canonical cache)

- **Схема данных (предложение, потом можно перенести в `database/models.py`)**:
  - `id`
  - `base_name` (строго нормализованное название на русском, без веса/бренда)
  - `display_name` (рекомендованное отображаемое имя, может включать бренд/вариант)
  - `category`
  - `source` (например: `"ai_norm_v1"`, `"label_ocr"`, `"manual"`)
  - `calories, protein, fat, carbs, fiber`
  - `per_unit` (по умолчанию `"per_100g"`, но можно `"per_piece"` и т.п.)
  - `version`, `created_at`, `updated_at`
- **Алгоритм `get_product_nutrition` (высокоуровнево)**:
  1. Нормализовать запрос (через лёгкий локальный нормализатор или существующий `NormalizationService`, но без КБЖУ — только имя/категория).
  2. Попробовать найти точное или почти точное совпадение в кэше по `base_name` (и, опционально, по бренду/категории).
  3. Если найдено — вернуть кэш как основной ответ, не обращаясь к ИИ.
  4. Если не найдено — вызвать существующий путь (ИИ/label/recog), получить КБЖУ, прогнать через anomaly guard, и если всё ок — записать в кэш и вернуть.
  5. Для повторных вызовов для того же продукта кэш становится источником правды.

#### 2. Перепроверяльщик (rechecker / auditor)

- **Задача**: "эталонизировать" старые/нестабильные данные, чтобы со временем весь зоопарк КБЖУ сходился к более чистому набору.
- **Варианты триггеров**:
  - фоновый джоб (по расписанию) — проход по топ‑N самых часто встречающихся `base_name` и сверка с текущим кэшем/новым ИИ.
  - ручной триггер для конкретного продукта (например, кнопка в админке/команде, либо сервисная команда в чате).
- **Алгоритм ретроспективной перепроверки**:
  1. Выбрать группу продуктов/логов с одним `base_name` но разными КБЖУ (расхождения выше порогов).
  2. Сформировать запрос к ИИ/веб‑поиску с жёсткими требованиями к формату (как в `NormalizationService`), но с акцентом на источники (официальные таблицы, FatSecret, USDA и т.п.).
  3. Полученный ответ прогнать через anomaly guard.
  4. Записать как новую версию эталона в кэш (старую пометить устаревшей, но не удалять).
  5. (опционально) Предлагать пользователю/админу миграцию старых записей (для отчётов/аналитики).

#### 3. Anomaly Guard

- **Примеры правил**:
  - Масла/жиры: калории < 600 ккал/100г → подозрительно.
  - Овощи: калории > 150 ккал/100г → подозрительно.
  - Белок в чистом мясе < 12 г/100г или > 40 г/100г → подозрительно.
  - Если есть кэш‑эталон: расхождение > X% по любой компоненте → флаг `WARN` или `RECHECK`.
- **Интеграция**:
  - Вызывается внутри нового модуля **до** записи в кэш и **до** логирования в питание.
  - Может возвращать сообщение‑подсказку для пользователя (в духе: "похоже на X, но цифры странные, хочешь перепроверить?").

### Интеграция с текущим кодом (первый этап)

- **Не трогаем** текущие хэндлеры массово (например, `handlers/i_ate.py`, `handlers/universal_input.py`), а добавляем:
  - новый модуль, который можно вызывать вместо прямых вызовов `NormalizationService.analyze_food_intake` и `AIService.recognize_product_from_image` в нескольких точках.
- **Места для первой интеграции** (минимальная инвазивность):
  - точка текстового логирования еды `process_text_food_logging` в `[handlers/universal_input.py](foodflow-bot_new/handlers/universal_input.py)`;
  - точка распознавания по фото еды в `process_photo_food_logging` в том же файле;
  - точка добавления продукта в холодильник по тексту `process_text_fridge_add`.
- **Стратегия**:
  - создать новый сервис и внутри него использовать **существующий** `NormalizationService/AIService` как "поставщиков сырых КБЖУ";
  - в первых местах интеграции вызывать новый сервис и сравнить поведение для ограниченного числа пользователей (например, только для админа/ID‑списка, контролируемым флагом).
